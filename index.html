ToDo: Minimal To Do List

After spending a significant amount of time working with React and Javascript on the front-end, Python and Javascript on the backend, and Postgres to make simple databases, I was still unaware how to package technologies together into a functional application. Building ToDo was my way of learning just that. Below is a short summary of some personal design choices and highlights from my learning along the way.

Relational Databases with PostgreSQL

                                                    
// Search todos
router.get("/todos/search/:search", 
    authorization, 
    async(req, res) => {
        try {
            const { search } = req.params;
            const user_data = await db.query(
                "SELECT u.user_name, t.todo_id, t.description 
                FROM users as u 
                LEFT JOIN todos as t 
                ON u.user_id = t.user_id 
                WHERE (u.user_id=$1 
                AND (description LIKE ('%' || $2 || '%'))) 
                ORDER BY t.todo_id asc",
                [req.user.id, search]
            );
            res.json(user_data.rows);
        } catch (error) {
            console.error(error.message);
        }
})
  • The pg module is used to access PostgreSQL and store both user profiles and ToDo list items in a relational way
  • The appropriate database queries are made at each endpoint, with variables defined by the client safely injected into the query by Postgres through the variable array
  • User IDs are stored as Universal Unique Identifiers which acts as the foreign key for each ToDo item
  • Passwords are salted and encrypted through the bcrpyt module before being stored, and subsequently validated by bcrypt upon login

JSON Web Token Authorization

                                                    
// Authorization
const jwt = require("jsonwebtoken");
require("dotenv").config();

module.exports = async (req, res, next) => {
    try {
        // Grab token from request header
        const jwtToken = req.header("token");

        // Check if there is no token given
        if (!jwtToken) {
            return res.status(403)
                .json("Not Authorized");
        }
        // Use jwt to verify the token with the secret key
        const payload = jwt.verify(
            jwtToken, 
            process.env.jwtSecret
        );

        // Add the user ID from the verified token's 
        // payload to the request, 
        // and send to the next destination  
        req.user = payload.user;
        next();
    } catch (error) {
        console.error(error.message);
        return res.status(403)
            .json("Authoization: Not Authorized");
    }
}
  • JSON Web Tokens are used to provide client access to authorized URLs and endpoint
  • A JWT is generated by the jsonwebtoken module and given to the client upon each successful register or login
  • The JWT is then provided back to the server in the header of each subsequent API request
  • The "authorization" middleware is then called prior to each endpoint, where the JWT is verified using the jsonwebtoken module

Ripe Potatoes: Movie Review Blog

Ripe Potatoes is my attempt at stepping into a project with a much larger-scale backend. The application is meant to be a movie review blog, taking advantage of the well-structured The Movie Database API. I took on this project to better understand how a more complex relational database should be structured, and how a backend API should interact with that database.

TypeScript Implementation

                                                    
import jwt, {Secret} from "jsonwebtoken";
import {Request, Response} from "express";
import dotenv from 'dotenv';
dotenv.config()

export default async (req: Request, res: Response, next: any) => {
    try {
        // grab token from request header
        const jwtToken = req.header("token");
        // check if there is no token given
        if (!jwtToken) {
            return res.status(403).json("Not Authorized");
        }
        // use jwt to verify the token with the secret key
        const secret_key: Secret = process.env.jwtSecret!;
        const payload = jwt.verify(jwtToken, secret_key) as {user: {id: string}};
        req.user = payload.user;
        next();
    } catch (error: any) {
        console.error(error.message);
        return res.status(403).json("Authoization: Not Authorized")
    }
}
  • Typescript is used to enhance the development process, improving testing and error checking
  • Module types are imported from the appropiate type libraries, which are installed via npm.
  • The larger scale of this proejct benefits from TypeScript. As file number increases, TypeScript's automatic primitive and method checker has significantly reduced debugging time.

Complex Relational Database

  • Users are liked to their respective reviews, comments, likes, and other interactions through their UUID foreign key
  • Movies from The Movie Database API are stored in the database by their URI at the time of first interaction with a user. The movies list will grow as interaction increases
  • Each type of user interaction has its own database table, so all interactions can be stored for efficient user history lookup

  • The database is built, but the majority of the backend endpoints still need to be built, this will likely be done after designing a larger portion of the front end to get a better idea of how we want our backend to function based on user interaction

Poke-GPT: AI Pokemon Battle Simulator

One of the projects in a coding course I took was to build a front-end only website that used a third-party API. Since I am overwhelmingly nerdy, I chose PokeAPI, a very impressive API that returns detailed information on the wonderful world of Pokemon.

I also had been hearing about the looming threat of our AI overlords, and thought it would be a good idea to get on their good side and implement a AI tool like OpenAI's ChatGPT API to my project. Thus I ended up with a single click interface that used PokeAPI to fetch the data on two random pokemon, and then used the OpenAI API to give that data to ChatGPT and request a narrated simulated Pokemon battle. I then decide to make a backend that dealt with the OpenAI API calls in order to hide my API key.

Third-Party APIs

                                                    
async function getResponseFromChatGPT(pokeNames, randMoves) {
    try {
        const apiKey = process.env.chatApiKey;
        const url = `https://api.openai.com/v1/chat/completions`;
        const body = JSON.stringify({
            "model": "gpt-3.5-turbo", 
            "messages": [{
                "role": "user",
                "content": `Narrate a short pokemon battle between...`
            }],
            "temperature": 0, 
        });
        const myHeaders = new Headers();
        myHeaders.append("Content-Type", "application/json");
        myHeaders.append("Authorization", `Bearer ${apiKey}`);

        const response = await fetch(url, {
            body: body,
            method: 'POST',
            headers: myHeaders,
        });
    
        const data = await response.json();
        return data.choices[0].message.content.trim();
    }
  • Asyncronous function are used with the await command to ensure promises from the API request are resolved before dealing with the data
  • Each third-party API returns different data structures, proper testing is needed to ensure one understands the data recieved under different conditions from the API
  • The "dotenv" package is used to hide authentication keys in the backend to ensure security
  • The Headers constructor is used to build a proper request that is expected by the OpenAI endpoint

        nameDiv.innerText = capName;
        spriteDiv.src = pokemon.sprites.front_default;
        cardDiv.style.backgroundColor = colors[pokemon.types[0].type.name];

        const typeElems = document.getElementById(`poke-type-${pokeIdx}`).children;
        for(i = 1; i < typeElems.length; i++) {
            typeElems[i].innerText = '';
        };

        pokemon.types.forEach((typeObj, typeIdx) => {

};

Science

Before entering the development world, I studied physical chemistry at McMaster University in Hamilton, Ontario. I completed my Ph.D from 2017 to 2023 with the final thesis being titled:

"Solid-State and Diffusional Nuclear Magnetic Resonance Investigations of Oxidatively Stable Materials for Sodium Batteries"

*note: if you think naming things in programming is hard, you haven't been in academia

If you would like some light reading material the thesis is available in full print here. In short, my research was focused on using a specialized type of MRI (similar to what is used to image human tissues) called nuclear magnetic resonance (NMR), to study the chemical interactions of sodium- and lithium-based batteries. The overarching goal was learn how these batteries operated on the molecular level, understand the failure mechanisms of the materials, and use that understanding to improve them.

In my time at McMaster I had the chance to learn a number of skills, from more specific experimental techniques like solid-state NMR, electrochemical impedance spectroscopy (EIS), and electron paramagnetic resonance (EPR), to more transferable scientific skills such as independent project design, large-set data analysis, collaborative working, and scientific communication.

These skills, and in particular my ability to work efficiently in self-directed and collaborative environments, were developed through the completion of a number of independent, and larger collaborative projects. In team environments, I had the opportunity to contribute both as a team member in multi-university collaborative studies, and as the project lead in international multi-disciplinary projects. Each of which have been published in major scientific journals.
